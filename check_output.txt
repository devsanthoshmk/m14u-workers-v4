# M14U — Premium Music Streaming Platform Frontend

Build a modern, premium, daily-driver music platform frontend that connects to the existing Express backend at `localhost:4000`. The design philosophy prioritizes psychological UX reasoning for every element, ensuring it's usable, smooth, and feels like a professional product users reach for daily.

---

## User Review Required

> [!IMPORTANT]
> **Design Direction**: I'm going with a **dark-first luxury aesthetic** — deep blacks with warm amber/gold accents (inspired by high-end audio equipment and vinyl culture). This avoids the "Spotify clone" look while maintaining music-platform familiarity. The psychology: dark themes reduce eye strain during evening listening (peak music hours), and warm accents create emotional warmth associated with music.

> [!IMPORTANT]
> **No Authentication**: The API has no auth endpoints. All features (queue, preferences, history) will be client-side via LocalStorage only. The listen-along feature you mentioned will export player state as JSON — the actual sync server is out of scope for now, but the state structure will be ready for it.

## Proposed Changes

### Design System & Aesthetic Direction

**Psychological reasoning for every major design decision:**

| Element | Decision | Psychological Reasoning |
|---|---|---|
| **Dark theme** | Deep charcoal `#0A0A0B` with warm tones | Reduces eye strain during evening listening (70% of music is consumed after 6pm). Dark UI makes album art "pop" as focus |
| **Accent color** | Warm amber/gold `#E8A838` | Warmth & premium feel — evokes analogue audio gear, vinyl warmth. More distinct than Spotify green or YouTube red |
| **Typography** | "Plus Jakarta Sans" (headings) + "DM Sans" (body) | Geometric sans-serifs convey precision/modernity while remaining highly legible at small sizes on mobile |
| **Card hover** | Subtle glow + slight scale | Communicates interactivity without jarring motion. Glow mimics warm light = inviting |
| **Bottom player bar** | Always visible, 72px on mobile, 80px on desktop | Persistent control = user never feels "lost" from their music. Spotify/YouTube Music pattern = zero learning curve |
| **Progress bar** | Thin line that expands on hover/touch | Reduces visual clutter at rest, invites interaction on engagement (Fitts's law) |
| **Queue panel** | Slide-in from right with backdrop blur | Spatial consistency — queue is "beside" the player, not replacing it. Blur maintains context |
| **Lyrics** | Center-aligned, current line highlighted + scaled | Karaoke UX pattern. Scale draws eye. Preceding/following lines dimmed = focus on now |
| **Sleep timer** | Circular countdown in the player dropdown | Borrowed from alarm clock mental model. Circle = time passing |

#### API Service Layer

##### [NEW] [api.ts](file:///home/santhoshmk/EDUCATION%20CONTENT/Projects/M14U-js/frontend/src/services/api.ts)
- `searchSongs(query: string): Promise<Song[]>` — calls `GET /api/search?q=`
- `getTrending(locale?: string): Promise<TrendingResponse>` — calls `GET /api/trending?q=`
- `getStreamUrl(videoId: string): Promise<string>` — calls `GET /api/stream/:videoId`
- Base URL from `.env` variable `VITE_API_URL` defaulting to `http://localhost:4000`

##### [NEW] [lyrics.ts](file:///home/santhoshmk/EDUCATION%20CONTENT/Projects/M14U-js/frontend/src/services/lyrics.ts)
- `getLyrics(trackName, artistName, albumName?, duration?)` — calls LRCLIB `GET /api/get`
- `searchLyrics(query)` — calls LRCLIB `GET /api/search?q=`
- Parses LRC format `[mm:ss.xx]` into `{ time: number, text: string }[]`
- Falls back to unsynced plain lyrics when synced unavailable

---

#### Player Engine

##### [NEW] [AudioEngine.ts](file:///home/santhoshmk/EDUCATION%20CONTENT/Projects/M14U-js/frontend/src/engine/AudioEngine.ts)
- Singleton class wrapping `HTMLAudioElement`
- Methods: `play()`, `pause()`, `seek(time)`, `setVolume(0-1)`, `loadSource(url)`
- All events (`timeupdate`, `ended`, `error`, `pause`, `play`, `waiting`, `canplay`) forwarded to callbacks
- Handles unexpected pauses (Bluetooth disconnect, tab visibility, error recovery)
- Pre-fetches next song stream URL when current song is 80% complete

---

#### Player Store (Zustand)

##### [NEW] [playerStore.ts](file:///home/santhoshmk/EDUCATION%20CONTENT/Projects/M14U-js/frontend/src/stores/playerStore.ts)
- **State**: `currentSong`, `isPlaying`, `currentTime`, `duration`, `volume`, `isMuted`, `queue[]`, `queueIndex`, `repeatMode` (off/one/all), `isShuffled`, `originalQueue[]`
- **Actions**: `playSong()`, `togglePlay()`, `next()`, `previous()`, `seek()`, `setVolume()`, `addToQueue()`, `addNext()`, `removeFromQueue()`, `reorderQueue()`, `clearQueue()`
- `exportState()` — returns JSON-serializable snapshot for listen-along sharing
- `importState(json)` — restores from a shared state
- Persisted to LocalStorage via Zustand `persist` middleware (excludes `currentTime` rapidly-changing values)
